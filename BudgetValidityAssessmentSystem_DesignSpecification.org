* 1. システム概要

** 1.1 目的

予算額妥当性評価システムは、日本の政府IT調達における以下の標準ガイドラインに準拠した
予算評価業務を支援するWebアプリケーションである。

- DS-110 :: デジタル・ガバメント推進標準ガイドライン（第3編第3章「予算及び執行」）
- DS-910 :: 安全保障等の機微な情報に係る政府情報システムの取扱い

評価者とPJMOの往復評価（ラリー）を構造化・履歴管理し、複数の評価対象システムおよびその
事業単位での評価進捗を一元管理する。

** 1.2 主要機能

| 機能                         | 説明                                                                                        |
|------------------------------+---------------------------------------------------------------------------------------------|
| 評価対象システム管理         | 複数の評価対象システムと担当部署を登録・管理                                                |
| *評価区分切替*               | システム単位で DS-110（政府情報システム）/ DS-910（省システム・機微情報）を切替             |
| 事業管理                     | システム配下の事業（全体仮想事業 + 1..N件）を管理。経費区分・要求額・見積・要件・根拠を保持 |
| 評価ラリー                   | 事業単位で評価者↔PJMOの指摘-回答サイクルを管理。複数サイクル対応                            |
| 予算評価AIアドバイザー       | Ollama（qwen2.5）によるDS-110/DS-910準拠の対話型評価支援                                    |
| AI自動レビュー               | テキスト貼付によるOllama自動指摘生成→スレッド自動作成。*評価区分に応じてプロンプトを自動切替* |
| Excel/PDFエクスポート        | 評価ラリーの監査ログを出力                                                                  |
| JSONエクスポート/インポート  | システム単位／事業単位／リレー履歴単位での移植・同期                                        |
| 監査ログ                     | 全操作（JSON I/O、AI指摘の評価区分を含む）を audit_logs テーブルに記録                      |
| MCP/Webhook通知              | Agent Card（JSON-LD）形式でのイベント通知                                                   |

** 2.3 systems（評価対象システム）

=ds910_applicable= フラグで評価区分を識別する。TRUE の場合、AI自動指摘のプロンプトが
DS-910 観点（機微情報・SBOM・国内立地・自律性等）に切り替わる。

#+begin_src sql
CREATE TABLE IF NOT EXISTS systems (
    id                SERIAL PRIMARY KEY,
    name              TEXT NOT NULL,        -- システム名
    department        TEXT,                 -- 担当部署
    ds910_applicable  BOOLEAN NOT NULL DEFAULT FALSE,
                                            -- TRUE:  DS-910対象（省システム・機微情報）
                                            -- FALSE: DS-110 準拠の政府情報システム
    created_at        TIMESTAMP DEFAULT now()
);

-- 既存DBへのカラム後追い（v2.0 → v2.1 マイグレーション）
ALTER TABLE systems
    ADD COLUMN IF NOT EXISTS ds910_applicable BOOLEAN NOT NULL DEFAULT FALSE;
#+end_src

*** 評価区分の判定基準

| ds910_applicable | 区分                 | 評価観点の根拠                                   |
|------------------+----------------------+--------------------------------------------------|
| FALSE            | 政府情報システム     | DS-110（予算妥当性・Cloud by Default・同等品条項）|
| TRUE             | 省システム・機微情報 | DS-910（機微情報範囲・SBOM・国内立地・自律性水準）|

DS-910対象となる機微情報の範囲は §Ⅴ（国家安全影響情報・国際交渉情報・公共安全情報・経済安全保障情報）に準じる。

** 2.4 projects（事業）

各システムには「システム全体」用の *仮想事業*（=is_system_wide=TRUE=）を1件自動生成する。
システム追加時に =systems_router.py= の create_system が自動挿入する。

#+begin_src sql
CREATE TABLE IF NOT EXISTS projects (
    id                 SERIAL PRIMARY KEY,
    system_id          INT REFERENCES systems(id) NOT NULL,
    name               TEXT NOT NULL,          -- 事業名
    is_system_wide     BOOLEAN DEFAULT FALSE,  -- TRUE: システム全体スレッド用の仮想事業
    cost_category      TEXT
                       CHECK (cost_category IN (
                           '整備費（投資的整備）',
                           '整備費（維持的整備）',
                           '整備費（サービス利用料）',
                           '運用費',
                           '保守費'
                       )),
    -- 要求額（千円単位）
    amount_single      INTEGER DEFAULT 0,      -- 単歳要求額
    amount_advance     INTEGER DEFAULT 0,      -- 前払金
    amount_deferred    INTEGER DEFAULT 0,      -- 後年度負担金
    -- 見積・要件・根拠
    estimate_text      TEXT,                   -- 見積（テキスト）
    estimate_file      TEXT,                   -- 見積（ファイルパス）
    requirements_text  TEXT,                   -- 要件（テキスト）
    requirements_file  TEXT,                   -- 要件（ファイルパス）
    basis_text         TEXT,                   -- 根拠（テキスト）
    basis_file         TEXT,                   -- 根拠（ファイルパス）
    evaluator_name     TEXT,                   -- 評価者名（自由入力・自動保存）
    created_at         TIMESTAMP DEFAULT now()
);
#+end_src

#+begin_quote
設計上の注意：
- システム全体スレッドは =is_system_wide = TRUE= の仮想事業配下に作成される
- システム削除は配下の =is_system_wide = FALSE= の事業が0件の場合のみ可能
- JSONインポート時は新規採番される（id は保持されない）
#+end_quote

*** 経費区分（別紙２準拠）

| 値                         | 別紙２区分                                           |
|----------------------------+------------------------------------------------------|
| 整備費（投資的整備）       | 整備経費 > 投資的整備（積極的に効果を得ることを目的とするもの） |
| 整備費（維持的整備）       | 整備経費 > 維持的整備（外部環境変更等による義務的整備）|
| 整備費（サービス利用料）   | 整備経費 > サービス利用料（クラウドサービス等）       |
| 運用費                     | 運用等経費 > 運用費                                  |
| 保守費                     | 運用等経費 > 保守費                                  |

** 2.5 review_threads（指摘スレッド）

#+begin_src sql
CREATE TABLE IF NOT EXISTS review_threads (
    id          SERIAL PRIMARY KEY,
    project_id  INT REFERENCES projects(id) NOT NULL,
                -- is_system_wide=TRUE の仮想事業配下: システム全体スレッド
                -- is_system_wide=FALSE の通常事業配下: 事業別スレッド
    thread_no   INT NOT NULL,     -- 事業内連番（#1, #2...）
    title       TEXT,             -- 指摘タイトル（任意）
    status      TEXT NOT NULL DEFAULT 'open'
                CHECK (status IN ('open', 'replied', 'reviewing', 'closed')),
    due_date    DATE,             -- 期限日（スレッド単位）
    created_at  TIMESTAMP DEFAULT now(),
    UNIQUE (project_id, thread_no)
);
#+end_src

** 2.6 review_entries（スレッド内投稿）

#+begin_src sql
CREATE TABLE IF NOT EXISTS review_entries (
    id          SERIAL PRIMARY KEY,
    thread_id   INT REFERENCES review_threads(id) NOT NULL,
    seq         INT NOT NULL,     -- スレッド内連番（1, 2, 3...）
    entry_type  TEXT NOT NULL
                CHECK (entry_type IN ('issue', 'reply', 'review', 'close')),
    role        TEXT NOT NULL
                CHECK (role IN ('evaluator', 'pjmo', 'ai')),
                -- 評価者・PJMOの個人名は保持しない
    comment     TEXT NOT NULL,
    entry_date  DATE,             -- 記録日付
    created_at  TIMESTAMP DEFAULT now(),
    UNIQUE (thread_id, seq)
);
#+end_src

#+begin_quote
設計上の注意：評価者・PJMOの個人名は =review_entries= に保持しない。
評価者名は =projects.evaluator_name= のみで管理する。
#+end_quote

** 2.7 ステータス遷移

*** スレッドステータス（review_threads.status）

#+begin_example
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  open ──[PJMO reply]──▶ replied ──[evaluator review]──▶ reviewing
│                                            │                │
│           ◀──[PJMO reply]──────────────────┘                │
│                                      （複数サイクル繰り返し可）│
│                                                             │
│  replied / reviewing ──[evaluator close]──▶ closed          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
#+end_example

*** entry_type と role の対応

| entry_type | role      | thread.status 変化         | 説明              |
|------------+-----------+----------------------------+-------------------|
| issue      | evaluator | → open                    | 指摘投稿          |
| issue      | ai        | → open                    | AI自動指摘        |
| reply      | pjmo      | open/reviewing → replied  | PJMO回答          |
| review     | evaluator | replied → reviewing       | 評価者再評価      |
| close      | evaluator | any → closed              | 確定（サイクル終了） |

*** ステータス説明

| status    | 説明                                        |
|-----------+---------------------------------------------|
| open      | 指摘が入った状態（PJMO未回答）              |
| replied   | PJMOが回答済み（評価者の再評価待ち）        |
| reviewing | 評価者が再評価中（PJMOの再回答待ち）        |
| closed    | 確定（評価サイクル完了）                    |

*** AI自動指摘の評価区分切替

=/threads/auto= は対象事業の所属システムの =ds910_applicable= を参照して
プロンプトを動的に選択する。

| ds910_applicable | system_prompt        | 生成スレッドタイトル    | レスポンス mode |
|------------------+----------------------+-------------------------+-----------------|
| TRUE             | DS-910観点プロンプト | =[AI自動指摘/DS-910]=   | =DS-910=        |
| FALSE            | DS-110観点プロンプト | =[AI自動指摘/DS-110]=   | =DS-110=        |

AIへの user メッセージ冒頭には以下の文脈ヘッダが自動付与される：

#+begin_example
【対象システム情報】
・システム名: {systems.name}
・担当部署: {systems.department}
・区分: {DS-910対象（省システム・機微情報）|政府情報システム（DS-110準拠）}

【評価対象ドキュメント】
{ユーザー貼付テキスト}
#+end_example

** 4.3 システム管理画面

- システム新規作成モーダル
  - システム名（必須）・担当部署
  - *DS-910対象チェックボックス*（機微情報を扱う省システムの場合にチェック）

- 各カードの操作ボタン
  - *⇄ 区分* : DS-110 ↔ DS-910 の評価区分切替（confirm プロンプト付）
  - *📤 出力* : システム一式を JSON ダウンロード
  - *🗑 削除* : システム削除（配下の非仮想事業が0件時のみ可）
  - *▼ 事業一覧* : 配下事業の展開
- 画面上部の *📥 インポート* ボタンで JSON ファイルからシステム取り込み（同名重複時は confirm プロンプト）
- 事業追加フォーム（経費区分・要求額3種・評価者名・見積/要件/根拠テキスト+ファイル）

** 6.2 インポート時の動作

- =system.id= は無視し、新規採番する
- =project.id= は無視し、新規採番する（is_system_wide の仮想事業も正しく復元される）
- =ds910_applicable= は設計書 §2.3 の評価区分としてそのまま取り込む
- スレッドと投稿は =thread_no= および =seq= の順序で復元する
- 既存データとの重複チェックは =/import/system/check= で事前に確認（§6.5）

リレーデータエクスポート（=/export/project/{id}/relay=）の出力形式：

リレーデータのインポートは mode パラメータで動作が変わる：

| mode      | 動作                                                           |
|-----------+----------------------------------------------------------------|
| =append=  | 既存スレッドを残し、thread_no を再採番して追記                 |
| =replace= | 既存の review_threads / review_entries を全削除してから投入    |

replace モードは破壊的操作のため =audit_logs= に削除件数と共に記録される。

** 6.5 インポート前重複チェック

システム単位インポート（=/import/system=）の前に、=/import/system/check= を呼ぶことで
同名システムの存在を確認できる。

{
  "duplicate": true,
  "matched": [
    { "id": 1, "name": "人事給与システム（次期）", "department": "人事院" }
  ],

  }

フロントエンドは =duplicate=true= の場合、=window.confirm= で利用者に確認を求める。
インポートは常に *新規採番* で実行され、既存システムは上書きされない。

* 7. 移行計画

** 7.1 v1.0 → v2.0 移行（実施済）

v1.0（PJMO予算評価レビューシステム）から v2.0（予算額妥当性評価システム）への
スキーマ移行は実施済みである。v1.0 由来の =backend/projects.py= / =backend/review.py=
は旧スキーマ参照のため =:tangle no= で生成対象から除外されており、参考資料として残置されている。

*** v1.0 → v2.0 の変換マッピング（記録）

| v1.0 テーブル/カラム            | v2.0 テーブル/カラム                        | 変換方法                         |
|---------------------------------+---------------------------------------------+----------------------------------|
| projects（category=system）     | systems                                     | systemsテーブルに移行            |
| projects（category=project）    | projects（新）                              | system_id を付与して移行         |
| projects.description            | systems.department                          | 担当部署として再利用             |
| review_logs（全体）             | review_threads + review_entries             | 1レコード → スレッド1件+エントリ1件 |
| review_logs.author              | （削除）                                    | 個人名は保持しない               |
| review_logs.version             | review_entries.seq                          | 連番として移行                   |
| review_logs.status              | entry_type + review_threads.status          | 分離して移行                     |
| review_logs.confirmed_by        | review_entries（entry_type='close'）        | closeエントリとして移行          |
| review_logs.confirmed_at        | review_entries.entry_date                   | 記録日付として移行               |

** 7.2 v2.0 → v2.1 移行（DS-910区分導入）

v2.1 では =systems.ds910_applicable= カラムを追加し、評価区分（DS-110/DS-910）を
システム単位で識別できるようにした。既存 pgdata を保持したまま列を追加する場合の手順：

#+begin_src shell :eval no
# 既存DBにカラム追加（init.sql にも同等の ALTER が同梱されている）
docker compose exec -T db psql -U budget -d budget <<SQL
ALTER TABLE systems
    ADD COLUMN IF NOT EXISTS ds910_applicable BOOLEAN NOT NULL DEFAULT FALSE;
SQL

# 既存システムを DS-910 対象へ切替える場合
docker compose exec db psql -U budget -d budget \
  -c "UPDATE systems SET ds910_applicable = TRUE WHERE id = 3;"
#+end_src

*** v2.0 → v2.1 の影響範囲

| 層       | 変更内容                                                                 |
|----------+--------------------------------------------------------------------------|
| DB       | =systems.ds910_applicable= 列追加（default FALSE）                       |
| Backend  | =/systems/= CRUD が区分読み書き対応、=/threads/auto= がプロンプト自動切替 |
| JSON入出力 | =ds910_applicable= をスキーマに含めて往復                                |
| Frontend | 新規モーダルのチェックボックス、カードのバッジ、区分切替ボタン、ラリー画面の評価モード表示 |
| 監査ログ | create_system / update_system / import_system / auto_thread に区分情報を記録 |

既存データは全て =ds910_applicable = FALSE=（DS-110評価）として扱われ、
機微情報システムに該当するものは管理者が個別に =UPDATE= するか、UI の区分切替で変更する。

** 8.3 AI自動指摘プロンプト設計（DS-110/DS-910切替）

=/threads/auto= は評価対象事業の所属システムの =ds910_applicable= を参照し、
以下2種類の system_prompt を切替える。いずれも根拠条項の明示・「一式」表記の排除・
「同等以上」条項の確認を求める点は共通である。

*** DS-910対象プロンプト（機微情報システム）

- 役割設定: 「日本の政府IT調達のセキュリティアーキテクト」
- 最優先評価観点:
  1. 機微情報の範囲と社会的影響の認識（§Ⅴ: 国家安全・国際交渉・公共安全・経済安全保障）
  2. 構成要素別リスク対応（§Ⅱ）
     - データ: 操作権限制限、暗号化、データ削除、冗長性
     - 運用: 運用体制、監査体制、障害対応
     - ソフトウェア: SBOM提出、脆弱性対応、ライフサイクル管理
     - ハードウェア: 調達リスク管理、ファームウェア管理
     - データセンタ・通信: 国内立地、SLA管理、冗長構成
  3. クラウド化評価（§Ⅲ: NISTクラウド要件、自律性確保、技術陳腐化リスク）
  4. データ連携評価（§Ⅳ: 自律性差異、連携脆弱性、データ保存・処理の所在）
  5. 自律性水準設定・国内外基準の継続注視（§Ⅰ）
- 副次評価: DS-110（予算妥当性・調達透明性）

*** DS-110対象プロンプト（政府情報システム）

- 役割設定: 「日本の政府IT調達のデジタル統括アドバイザー」
- 評価観点（DS-110 第3編第3章「予算及び執行」準拠）:
  1. 予算マネジメント基礎: コスト分解、当初想定と実運用の乖離、調達方法の事前検討
  2. 費用対効果・投資妥当性: システム機能範囲、中長期投資計画、国庫債務負担行為
  3. 資料準備・要求根拠の合理性: サービスレベル、実データ根拠、要求額の積算精査
  4. 経費見積りの適正性: 積算内訳、LCC、システム単位積算、経費区分、複数見積比較、*Cloud by Default*
  5. ガバナンス・承認プロセス: PMO資料提出、デジタル庁指摘対応、プロジェクト計画反映
